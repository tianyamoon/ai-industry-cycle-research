# AI 产业链研究仓库规则

## 一、总原则

本仓库的最高工作准则有两条，且顺序固定：

1. `第一性原理`
2. `方法论先于结论`

所谓第一性原理，在这个项目里指的是：

1. 不从结论出发倒推论据，而是从最基础、最可验证的事实上推演结论。
2. 不被市场情绪、流行叙事、事后归因牵着走，而是拆解驱动产业与股价变化的底层变量。
3. 不满足于“大家都这么说”，而要持续追问：
   - 到底是什么变量在变化？
   - 这个变量如何传导到产业、利润、估值和股价？
   - 哪个环节是因，哪个环节是果？
   - 这个判断能否被数据、价格、财报、政策或市场行为验证？

所谓方法论先于结论，指的是：

1. 如果研究框架错了，资料越多，偏差可能越大。
2. 做产业周期研究前，必须先校准研究镜头，而不是直接进入结论生产。
3. 本仓库默认要求参考顶级买方、卖方和长期投资研究框架，对自己的分析路径做外部校验。
4. 任何重要研究结论，都应当能同时经受住“第一性原理”与“成熟方法论”的双重检验。

## 二、项目定位

本仓库用于 AI 产业周期研究。方法上，以 2015-2025 年中国新能源产业周期为历史样本，复盘产业从概念导入、产业验证、主升扩散、供需失衡到龙头分化与退潮的完整过程，并把其中可迁移的方法映射到 AI 产业链研究。

本项目的目标不是堆资料，而是建立一套：

1. 可验证
2. 可追溯
3. 可复用
4. 可迁移

的产业周期研究方法。

## 三、三层结构理解

本仓库当前按三层研究生产链组织研究内容：

1. `过程层`
   - 研究方法论与证据生产
   - 负责历史复盘、时间轴资料、硬数据补证、图表素材
2. `桥接层`
   - 标准行业研究卡
   - 先做研究骨架，再决定是否升级成正式成稿
3. `结果层`
   - 顶级基金风格的最终输出组织
   - 负责正式总结、阶段上收、总表整合与正式正文

另外保留两个辅助层：

1. `00_总控台`
   - 负责方法论、状态源、模板、规则、总映射索引
2. `99_交接包`
   - 负责阶段留痕、接管入口与历史归档

## 四、Codex 在本仓库中的角色

Codex 在本仓库中应扮演“研究助理 + 资料整理员 + 结构化编辑”，不是自由发挥的写稿机器人。

默认工作原则：

1. 先读框架、索引、模板，再修改内容。
2. 凡属 `入库 / 建卡 / 新增页面 / 新增文件`，第一步必须先 `查重`：先搜索本库、附近目录、总表、索引、`99_交接包` 与关联 wiki 中是否已有同主题主文件、同名实体页或可承接的现有页面；若已有，优先回写，不重复新建。
3. 优先复用现有目录、总表、模板、索引，不轻易平行新建。
4. 所有重要判断尽量绑定证据，不写无依据结论。
5. 对不确定信息必须显式标注，不把猜测伪装成事实。
6. 缺数据时先保留结构与占位，方便后续补证。
7. 遇到复杂问题时，先拆成底层变量，再组织结论。
8. 遇到方法论问题时，先对标外部成熟框架，再推进具体研究。
9. 文件组织默认追求“少入口、少重复、少增量文件”；若现有文件能承接，就直接回写，不为局部方便额外拆新文件。
10. 新增文件不是禁止项，但必须满足“现有结构无法清楚承接”或“新增后明显提升导航与复用性”之一，且数量保持最小。
11. 每次完成一轮明确修改后，默认执行 `git add`、`git commit`、`git push`；若因冲突、远端或网络问题无法完成，必须明确说明阻塞原因。
12. 本仓库是公开研究仓库；公众号/自媒体稿、私域群聊原文或截图、个人持仓与自选池、带本机账户路径的材料不得提交。私域证据只保留脱敏后的待核变量与公开来源回证结果。

## 七、外部方法论对标要求

本仓库默认要求对标以下几类外部研究方法：

1. `宏观周期模板`
   - 看生产率、信用、利率、流动性、长短债务周期
2. `市场周期与心理周期`
   - 看乐观、拥挤、估值扩张、泡沫化、出清与反身性
3. `产业/公司生命周期`
   - 看导入、成长、成熟、震荡、衰退，以及 ROIC、WACC、现金流结构
4. `资本开支周期`
   - 看扩产、融资、供给释放、过剩、价格战、出清
5. `结构性长期力量`
   - 看人口、技术、地缘、能源转型、金融结构变化
6. `预期与估值框架`
   - 看市场价格里已经隐含了什么，而不只是基本面“看起来好不好”

## 八、第一性原理在本项目中的具体用法

做任何研究时，优先把问题拆成以下层次：

1. `事实层`
   - 发生了什么
   - 什么时候发生
   - 哪些数字或价格在变化
   - 哪些政策、财报、订单、产能、持仓能够证明它
2. `变量层`
   - 需求是否变化
   - 供给是否变化
   - 成本是否变化
   - 价格是否变化
   - 资本开支是否变化
   - 资金参与度是否变化
3. `传导层`
   - 变量如何传导到产业景气
   - 如何传导到公司收入、利润、现金流、估值
   - 如何传导到股价表现和风格分化
4. `验证层`
   - 股价是否提前反映
   - 财务是否随后兑现
   - 市场叙事是否与数据一致
   - 历史事实是否支持这个解释
5. `迁移层`
   - 这个历史规律是否能迁移到 AI
   - AI 当前所处阶段对应新能源历史的哪个阶段
   - 哪些变量在 AI 中有一一映射关系

如果一个判断无法通过上述链条表达清楚，就说明它还不够扎实。

## 九、研究方法约束

1. 先看市场表现，再解释产业逻辑，不能先写结论再反向找股价配合。
2. 要区分产业周期、盈利周期、价格周期、资本开支周期、资金参与周期。
3. 要把“事实”和“判断”分开。
4. 重要结论要能回溯到原始资料、数据表、时间轴或结构化汇总。
5. 失败案例、阴影资产、纯题材链条也要保留，它们是研究方法的一部分。
6. 所有判断都要尽量还原当时市场的信息环境，而不是只用今天的后视角解释历史。
7. 任何阶段判断都要考虑是否存在估值泡沫、拥挤交易、信用收缩或资本开支反噬。

## 十、证据标准

对单个公司、产业链条、阶段总结或总表类文件，尽量覆盖以下信息：

1. 样本定位：它代表哪一类周期角色
2. K线阶段节点：启动、加速、见顶、回撤、二次冲顶、阴影、修复或失败
3. 价格锚点：相关上游或下游价格变量
4. 财务拐点：收入、利润、毛利率、现金流、CapEx、存货、减值、ROIC、订单等
5. 新闻或政策原文：市场当时在交易什么叙事
6. 当前状态标签：如 `穿越`、`半穿越`、`周期承压`、`阴影`
7. AI映射一句话：这一历史样本对 AI 产业链有什么启发

如果缺数据：

1. 不得臆造数据。
2. 必须明确写出 `待补`、`待核`、`待确认`。
3. 优先保留结构与字段，方便后续实填。

## 十一、来源与引用规则

1. 日期尽量精确到具体时间点。
2. 单位必须写清楚，如 `亿元`、`万吨`、`元/吨`、`GW`、`GWh`、`PE`、`PB`。
3. 如果是时点数据，注明对应期间，如 `2021Q4`、`2023年末`、`2024-06`。
4. 如果引用的是归纳表述，不要夸大原始材料的确定性。
5. 能写来源名称时尽量写来源名称，能放链接时尽量放链接。
6. 多个来源冲突时，保留差异并采用更保守的表述。
7. 引用外部方法论时，应尽量说明它解决的是哪一类问题，而不是把它当成万能框架。

## 十二、写作规范

1. 研究内容默认使用中文，除非目标文件本身就是英文语境。
2. 优先使用清晰标题、要点、表格和索引，不要写成大段无结构散文。
3. 先用一句话给结论，再补事实依据。
4. 文档信息密度高时，建议主动拆分为 `事实`、`判断`、`待补点`。
5. 风格上追求简洁、可核验、可续写，不追求花哨表达。
6. 文件命名要贴近相邻文件风格，优先保持 `稳定主文件名 + 清晰职责后缀` 一致；`模板`、`索引`、`待补表`、`实填` 等职责标签可以保留，但不再把 `第一版`、`第二版` 作为活文档常态。

## 十三、文件与命名规则

1. 顶层目录已固定为 `00_总控台 / 01_过程层 / 02_桥接层 / 03_结果层 / 99_交接包`，未经明确要求，不要再次改顶层分层。
2. 优先更新已有索引、模板、总表，不要平行新建重复文档。
3. 新文件应尽量沿用所在目录的命名习惯，并优先复用已有稳定文件名。
4. 现有活文档继续迭代时，默认直接更新原文件，不再平行新建 `第一版`、`第二版`、`第三版`；若属于周期输出，可保留日期，但不保留 `第几版`。
5. 历史演进默认交给 Git 和 `90_历史归档 / 90_执行留痕`，不要在活跃目录同时保留多个版本号文件。
6. 如果新增文件会影响模块导航，应同步更新最近的 `README.md` 或索引文件。
7. 新建文件前，先判断是否可以通过“回写现有主文、补充现有总表、更新现有索引、归入 `90_*`、归入 `tmp/`”解决；只有这些路径都不合适时，才新增文件。
8. 同一主题默认只保留一个当前主文件；确需并存多个文件时，必须有清晰角色分工，如 `主文 / 执行留痕 / 历史归档 / 模板 / 待补表`，不能只是内容相近的平行副本。
9. 如果拆文件会让阅读路径变长、入口变多、维护成本上升，而收益不明显，则优先不拆。

## 十四、任务执行流程

1. 若任务涉及 `入库 / 建卡 / 新增页面 / 新增文件`，先做查重：检查本库、附近目录、总表、索引、`99_交接包` 与关联 wiki，确认没有同主题现有主文件、重复实体页或可直接承接的既有页面。
2. 识别目标层级、目标模块和目标产物。
3. 阅读附近上下文、模板、索引和已有成果。
4. 如任务涉及框架、阶段判断、行业映射或重大结论，先阅读方法论文档。
5. 判断当前任务属于哪一类：研究框架搭建、证据回填、K线图谱补全、总表整合、AI映射、正文写作、图表整理。
6. 先拆底层变量，再组织证据和结论。
7. 尽量复用已有结构完成最小必要修改。
8. 如改动影响多个文件的术语、阶段划分或标签，要检查一致性。
9. 若任务涉及文件新增，先做一次“是否必须新增”的判断；若不是明显必要，优先压缩到已有文件中完成。
10. 一轮修改完成后，默认把本轮变更整理成可提交状态，并提交推送，避免长期累积未提交改动。

## 十五、交接规则

本仓库依赖明确交接来保持研究连续性。

当一次会话对某个主题形成了明显推进，且用户需要阶段留痕时，应考虑在 `99_交接包/` 中补充交接材料。

交接内容至少应包括：

1. 本次完成了什么
2. 修改了哪些文件
3. 还有哪些未解决问题
4. 还有哪些证据缺口
5. 下一步最建议推进什么
6. 当前阶段判断依赖了哪些方法论镜头

## 十六、禁止事项

1. 不得编造股价、价格、财务数据、政策内容、新闻出处或时间点。
2. 不要把多个周期阶段混成模糊的宏观判断。
3. 不要为了“看起来完整”而删除 `待补` 占位。
4. 不要用今天的后视角替代当时市场真实的认知环境。
5. 不要轻易写出与 `00_总控台/研究框架.md` 冲突的平行框架。
6. 不要用空泛结论覆盖已有的细节证据。
7. 不要把“题材热度”直接当成“产业逻辑”。
8. 不要把“股价涨了”直接当成“基本面验证”。
9. 不要把单一方法论强行套到所有行业与阶段。
10. 不得提交公众号/自媒体成稿、私域原始材料、个人组合信息或本机绝对路径。

## 十七、优先级判断

1. 先补索引和总表，再扩写长文。
2. 先补证据结构，再润色表达。
3. 先明确不确定性，再尝试下结论。
4. 先还原历史市场视角，再做事后总结。
5. 先拆底层驱动变量，再谈宏观叙事。
6. 先做方法论校准，再做结论输出。
7. 先做可复用结构，再做一次性文字。

## 十八、完成标准

一个任务通常在满足以下条件时算完成：

1. 目标内容已写入现有仓库结构中。
2. 命名和格式与所在模块保持一致。
3. 关键判断有事实、数据、时间点或明确假设支撑。
4. 缺失项被清晰标注，而不是被隐藏。
5. 结论能被还原为“事实 -> 变量 -> 传导 -> 验证 -> 映射”的链条。
6. 结论经过了至少一种外部成熟方法论镜头的校验。
7. 如果影响导航，相关索引或 `README.md` 已同步更新。
8. 没有为非必要目的扩张文件数量；若新增文件，理由和角色是清楚的。
9. 本轮改动已完成提交与推送；若未完成，阻塞原因已明确记录。

如果因为资料不足无法完全完成任务，也应把结构搭好、缺口写明、下一步补证方向写清楚。
